# MemoDoc - 테스트 및 검증 계획 (TEST_PLAN)

---

## 가설

### 핵심 가설
> 업무 메모를 MemoDoc에 입력하면, 사람이 직접 정리하는 것보다 빠르고 실용적인 수준의 문서를 생성할 수 있다.

### 세부 가설 (검증 가능한 단위로 분해)

| ID | 가설 | 측정 방법 |
|---|---|---|
| H1 | 메모 입력 후 30초 이내에 첫 번째 탭 문서가 완성된다 | 생성 시작~완료 시간 측정 |
| H2 | 생성된 문서를 Notion에 붙여 넣었을 때 별도 수정 없이 사용 가능하다 | 사용자 수정 횟수 측정 |
| H3 | 메모 유형(회의/교육/아이디어)에 따라 요약 형식이 적절하게 달라진다 | 유형별 출력 섹션 일치율 확인 |
| H4 | Mermaid 다이어그램이 프로세스가 있는 메모에서 오류 없이 렌더링된다 | 렌더링 성공/실패 횟수 집계 |
| H5 | 사용자가 직접 문서를 작성하는 것보다 시간이 단축된다 | MemoDoc 소요 시간 vs 직접 작성 시간 비교 |

### 가설 우선순위 및 의존 관계

**우선순위:** H1(속도) > H5(시간 절감) > H2(호환성) > H3(유형별 분기) > H4(Mermaid) > H6(재사용) > H7(입소문)

**의존 관계:**
- H1 → H5: H1이 달성되어야 H5(전체 시간 절감)가 의미 있음
- H2 → H5: H2(품질)가 달성되어야 시간 절감이 실질적 가치를 가짐
- H3, H4는 독립적으로 검증 가능
- H5 → H6 → H7: 시간 절감이 확인되어야 재사용이 발생하고, 재사용이 있어야 입소문 전파

### 비즈니스 가설 (H6~H7)

| ID | 가설 | 측정 방법 | 실측값 |
|---|---|---|---|
| H6 | 첫 사용 후 재사용 의향이 80% 이상이다 | 검증 세션 후 재사용 의향 Y/N | 5/5명 (100%) ✅ |
| H7 | 한 명이 사용하면 팀원에게 공유하려는 의향이 생긴다 | 검증 세션 후 공유 의향 Y/N | 4/5명 (80%) ✅ |

## 검증 방법

### 1. 기능 검증 (개발자 테스트)
- 정의된 테스트 시나리오를 직접 실행하여 기능 동작 확인
- 다양한 메모 유형 및 길이로 반복 테스트

### 2. 사용자 검증 (유저 테스트)
- **대상:** 소프트웨어 개발자 또는 기획자 3~5명
- **방법:** 실제 업무 메모를 제공하고 MemoDoc 사용 후 피드백 수집
- **측정 항목:**
  - 생성 문서 품질 만족도 (5점 척도)
  - 문서 수정 없이 바로 사용 가능 여부 (Y/N)
  - 직접 작성 대비 시간 절감 체감 (5점 척도)
  - 가장 유용한 탭 / 가장 불필요한 탭 선택

### 3. 정량 측정
- 탭별 생성 소요 시간 (ms 단위, `performance.now()` 기록 → 콘솔 출력)
- Mermaid 렌더링 성공률 (성공 횟수 / 전체 시도)
- 오류 발생 시 재시도 성공률

## 사용자 피드백 수집 계획 (Phase 2)

### 설문 항목 (5점 척도)
1. 생성된 문서의 품질이 실무에서 바로 사용할 수 있는 수준인가?
2. 직접 정리하는 것보다 시간이 절약되었는가?
3. 6개 탭 중 실제로 유용하게 사용할 탭은 몇 개인가?
4. 다시 사용할 의향이 있는가?

### 정성 피드백 항목
- 가장 유용했던 탭과 그 이유
- 가장 아쉬웠던 점
- 추가되었으면 하는 기능

## 검증 결과 요약

### 확인된 가설
- **H1 (속도):** 첫 탭 15~25초로 30초 이내 목표 달성 ✅
- **H3 (유형별 분기):** 회의/교육/아이디어 유형별 섹션 구조 정확히 분기됨 ✅
- **H4 (Mermaid):** 8건 중 7건 오류 없이 렌더링 (87.5%) ✅
- **H5 (시간 절감):** 수동 25~35분 → MemoDoc 2~3분, 약 85~90% 절감 ✅
- **H6 (재사용):** 검증 세션 5건 모두 재사용 의향 표시 (100%) ✅
- **H7 (입소문):** 5건 중 4건 팀원 공유 의향 표시 (80%) ✅

### 부분 확인
- **H2 (Notion 호환성):** 5건 중 4건 수정 없이 활용 가능. 1건은 마크다운 헤딩 레벨 조정 필요

### 엣지케이스 테스트 결과

| 케이스 | 입력 | 결과 | 비고 |
|---|---|---|---|
| 장문 메모 (5,200자) | 기술 스펙 문서 전체 | 6탭 모두 생성 완료 (3분 10초) | document 탭 일부 잘림, 핵심은 보존 |
| 한영 혼용 메모 | "today 회의: API design 논의, REST vs GraphQL 결정" | 6탭 모두 정상 생성, 한국어 출력 유지 | ✅ |
| 최소 메모 (10자) | "오늘 회의." | 입력 차단 (10자 미만 경고) | ✅ 검증 통과 |
| 특수문자 포함 | `<script>`, `"quotes"`, `&ampersand` | XSS 없이 정상 렌더링 | marked.js가 자동 이스케이프 |
| 한 단어 반복 | "회의 회의 회의..." × 500 | 요약: "반복적 내용만 있습니다" 수준 출력 | 모델 품질 한계 |

### 추가 데이터 필요
- 외부 사용자 정식 설문 (현재 관찰 세션 5건 수행)
- 10,000자 최대 메모에서의 품질 측정

### 핵심 인사이트
1. 단일 메모로 6가지 관점의 문서를 동시 생성하는 것이 핵심 가치 — 개별 요청 대비 컨텍스트 전환 비용 없음
2. 무료 모델 한계로 생성 시간이 2~3분으로 길지만, 수동 작업 대비 압도적 절감
3. Mermaid 자동 생성이 가장 차별화된 기능 — 수동 작성 시 10~15분 소요를 자동화

### Phase 2 반영 사항
- 유료 모델 지원 → 병렬 처리 복원으로 전체 시간 20~30초로 단축 가능
- 외부 테스터 설문 수집 후 H2 재측정
- 장문 메모 요약 품질 개선 프롬프트 연구
